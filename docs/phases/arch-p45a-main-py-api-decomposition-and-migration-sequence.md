# ARCH-P45A - Target API Decomposition and Migration Sequence for `main.py` Split

Issue: `#774`  
Date: `2026-03-24`  
Scope: planning/documentation only. No runtime changes.

## Goal

Define the bounded target decomposition of the current API entry surface so `src/api/main.py` can be reduced safely without behavior drift.

## Constraints

- This issue does not split code or move runtime logic yet.
- This issue does not change endpoint behavior, payloads, status codes, lifecycle hooks, auth rules, or `/ui` mounting.
- `src/api/**` and `tests/**` remain unchanged in this issue.
- Output of this issue must be specific enough to create implementation issues.

## Bounded Target Decomposition

The target is strictly an API decomposition, not a broader architecture redesign.

### A. `main.py` end-state responsibility (composition root only)

`src/api/main.py` should retain only:

- app creation and metadata.
- static `/ui` mount.
- startup/shutdown registration.
- app-level state initialization required by existing behavior.
- router inclusion/wiring.

`main.py` should not retain endpoint handlers, API DTO class definitions, or domain orchestration helpers once migration completes.

### B. Router modules (`src/api/routers/*`)

Target router groups:

- `health_router.py`: `/health*` transport wiring.
- `control_plane_router.py`: `/runtime/introspection`, `/system/state`, `/execution/*`.
- `compliance_router.py`: compliance guard status.
- `portfolio_router.py`: portfolio and paper inspection reads.
- `trading_core_read_router.py`: trading-core and execution read endpoints.
- `analysis_router.py`: `/strategy/analyze`, `/analysis/run`.
- `screener_router.py`: `/screener/basic`, `/screener/v2/results`.
- `watchlist_router.py`: watchlist CRUD + execute.
- `inspection_router.py`: ingestion runs, signals, strategies, journal, decision-cards.
- `alerts_router.py`: existing alerts surface (can remain generated by builder function, but registered as dedicated router module contract).

Router scope is transport-only: request binding, dependency injection, response typing, and HTTP exception translation.

### C. Services (`src/api/services/*`)

Target service groups:

- `health_service.py`: runtime/data/guard health payload composition.
- `control_plane_service.py`: runtime/system state reads and execution lifecycle actions.
- `compliance_service.py`: compliance guard aggregation.
- `portfolio_service.py`: portfolio/paper account/trades/positions/reconciliation reads.
- `trading_core_read_service.py`: execution/trading-core reads.
- `analysis_service.py`: manual analysis + strategy analyze orchestration.
- `screener_service.py`: screener execution + results reads.
- `watchlist_service.py`: CRUD + execute orchestration and run reuse.
- `inspection_service.py`: ingestion runs, strategies, signals, journal, decision-card inspection.
- `alerts_service.py`: alert configuration/history operations.

Service scope: orchestration and composition only; no transport declarations.

### D. Builders/helpers (`src/api/builders/*`)

Builders are introduced to isolate deterministic assembly and formatting logic currently embedded in route files/services:

- `ranking_builder.py`: symbol and watchlist ranked results.
- `compliance_builder.py`: guard status payload assembly.
- `paper_account_builder.py`: account/reconciliation summary/mismatch construction.
- `inspection_builder.py`: decision-card extraction, sorting, filtering, and journal content shaping.

Builder scope: pure assembly/transform logic, deterministic output, no I/O side effects beyond inputs provided by service layer.

### E. DTO/model modules (`src/api/models/*`)

Target model partition:

- `analysis_models.py`
- `watchlist_models.py`
- `screener_models.py`
- `control_plane_models.py`
- `health_models.py`
- `compliance_models.py`
- `portfolio_models.py`
- `trading_core_read_models.py`
- `inspection_models.py`
- `alerts_models.py`

Scope rule: API transport DTOs and query models only.

### F. Dependencies (`src/api/dependencies/*`)

- `auth.py`: role enforcement and policy mapping.
- `query.py`: reusable query binders/parsers.
- `runtime.py`: shared runtime preconditions (for example engine-running guard).

## Migration Sequence and Issue Boundaries

Each step is intended to be one implementation issue with strict behavior preservation.

1. Extract DTO/query models from `main.py` to `src/api/models/*` with import-only rewiring.
2. Extract shared dependencies (`_require_role`, query binders, runtime preconditions) to `src/api/dependencies/*`.
3. Extract pure builders (`ranking`, `compliance`, `paper`, `inspection`) without changing call flow.
4. Introduce read-only services (health, compliance, portfolio/paper, trading-core read, inspection reads) and delegate from existing handlers in place.
5. Move read-only handlers to router modules and include routers from `main.py`.
6. Introduce mutation/execution services (analysis, screener execution, watchlist execute/CRUD, alerts mutations) and delegate from handlers in place.
7. Move mutation/execution handlers to router modules and include routers from `main.py`.
8. Remove obsolete helper/DTO definitions from `main.py` after all imports and routes are migrated.
9. Final consolidation issue: verify `main.py` contains only composition-root responsibilities.

## Behavior-Preservation Rules (Mandatory for every follow-up issue)

- No route path or HTTP method changes.
- No auth header/role semantic changes (`X-Cilly-Role` behavior preserved).
- No request/response schema changes.
- No status code changes (including error detail codes currently returned).
- No startup/shutdown or runtime control semantic changes.
- No `/ui` mount behavior changes.
- No change to deterministic run-id reuse behavior for analysis/watchlist execution.
- No data ordering/pagination behavior changes for read endpoints.
